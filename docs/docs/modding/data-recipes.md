# Data Recipes

:::note[Community-reported]
Concrete gameplay-tuning recipes sourced from the OWG forum and Discord communities — see
[Getting Started](./getting-started.md) for the full provenance note.
:::

Concrete data recipes found by the community:

- **Full-auto AR-16/M16**: change all `iBurstLength` values from `3` to `0` in `41_WeaponProperties.xml` (there are multiple instances per weapon — use find-all), copy the *singleplayer* copy of the weapon (search `hidName` without the `.Multi` suffix) into `mymod\patch\21_WeaponProperties.xml`, add the file to `patch.xml`'s object list below the existing `22_weapons.xml` entry. The "correct" underlying mechanism is the `selFireRateMode` enum — flipping that directly to `FullAuto` is more "correct" but silences the weapon's firing sound as a side effect (confirmed: this is `iBurstLength`'s hard-coded relationship to the sound trigger, not a bug specific to one weapon), so the community stuck with the `iBurstLength` workaround for audio-safe results.
- **Shotgun pellet count**: `iBulletsShot` (e.g. USAS-12) — raising from 7 toward ~20 turns it into a one-shot kill machine; going much higher risks performance slowdown.
- **Accuracy / bullet spread**: `bUseAngleSpread` defaults `False` on rifles (shotguns default `True`) — set `True` and tune `fAngleYawBulletSpread`/`fAnglePitchBulletSpread` (degrees). Whole numbers like `1` are still far too inaccurate; realistic spread needs small decimals (e.g. `0.002`). Conversion formula from a real-world group size to the in-game angle: **`2 * atan((GroupSizeInches/2) / 36000)`** (for a group size at 1000 yards). Weapon recoil lives in `42_weapons.xml`'s `<ReliabilityLevelsData>` block (see jam mechanics below), not in WeaponProperties.
- **Damage / hit locations**: base damage is **not** controlled by `gamemodesconfig.xml`'s cosmetic `<Summary>` block — it's in `41_WeaponProperties.xml`. Add `selHitLocation_Torso_Severity`/`selHitLocation_Limb_Severity` (UInt32, copy from an existing sniper-rifle entry) near `fHealthFailureChanceModifier` to get realistic one-shot-drops. A separate `Stim_ImpactDamage` block's `nLevel` sets a weapon's damage tier (reference values found: MAC10=7, FAL=14, Dragunov=25, AS50=30).
- **Recoil / the "42_weapons.xml crash" workaround**: directly overriding `42_weapons.xml` in the patch folder **crashes the game on load**, every time, confirmed reproducible — renaming it doesn't help. The only working method: copy the target weapon's entire data block out of `42_weapons.xml` (in `world1`/`world2`) and paste it **into the existing corresponding weapon section of `22_weapons.xml`** (the file that's normally MP-only) inside `mymod\patch`. This "smuggles" SP-relevant changes (recoil, jam probability, etc.) through a file the patch system will actually accept. Must be repeated per weapon.
- **Jam & reliability**: `42_weapons.xml`'s `<ReliabilityLevelsData>` block has `Failure`/`Low`/`Medium`/`High` sub-objects, each with `fHorizontalRecoilPerShot`, `fVerticalRecoilPerShot`, `fBulletDeviationMax`, and `fJamProbabilityPerReload` (e.g. Failure=0.12, Low=0.06, Medium=0.03, High=0) — this is the real jam-probability mechanism, tied to a reliability tier that itself is driven by `gamemodesconfig.xml`'s per-Act `CounterReliabilityRatingsTable` thresholds. `iClipsForSelfDestruct` (low value = degrades+jams faster, but degradation and self-destruct are **separate** mechanisms, not one linear scale) and `bIsIndestructible`/`bIsBreakable` also apply — but a weapon set indestructible will still show cosmetic "rust" over time (functional and cosmetic wear are separate systems). The M-79's "jam" is not a true jam — it's a grenade round physically misfiring at short range.
- **Magazine capacity** (long unsolved, cracked in 2012): in `41_WeaponProperties.xml`, search the hash `4FBDD114` to jump to any weapon's ammo block. Confirmed hash → meaning map:
  - `AB258E09` (BinHex) — ammo type string, e.g. `"assaultrifle\0"`
  - `88596C97` (BinHex) — `iAmmoInClip` (magazine size)
  - `2A0F1CC2` — `iMaxAmmoCasual`
  - `C7DA96EA` — `iMaxAmmoExperimented`
  - `EF3C58C3` — `iMaxAmmoHardcore`
  - `DE33B3EC` — `iMaxAmmoInfamous`
  - a separate bool-like value = "ammo visible on HUD" (true only for belt-fed PKM/M249 — controls whether the visible ammo belt depletes)
  - `iClipsForSelfDestruct` ties weapon-breaking to **magazines fired**, not total shots fired — must be adjusted inversely if you change magazine size.
  - **Gotcha**: these are raw little-endian hex (`BinHex` type). Easiest fix: manually change the value's `type` attribute from `BinHex` to `UInt32` (or `Bool`) in the XML before running `ConvertBinary` — it will then accept plain decimal/bool and repack correctly.
  - `ConvertBinary` gives no line numbers on malformed XML, just a raw stack trace — re-run it after every small edit to isolate mistakes.
- **Small ammo/explosive/fuel pickups** (`28_pickups.xml`, world1): each "Ammo.Small_X_Pickup" block has per-weapon-category `AmmoEntry` objects keyed by hash pairs (hash / little-endian BinHex). Confirmed tables:
  - Ammo: `6D6540FA`=Pistols(Star45), `BC6782FC`=Assault Rifles(AK47), `7D6BD5F2`=Sniper Rifles(M1903), `EEAE53E1`=Shotguns(SPAS12), `AA73EE0A`=SMGs(MAC10), `BD090A47`=LMGs(PKM), `FC2096BC`=Dart Rifle
  - Explosive: `4EE9BFD6`=Mortars, `CEB9BB1E`=RPG-7, `704CA95D`=M-79, `EA12131E`=IED, `E710123D`=MGL-140 (+ separate grenades category)
  - Fuel: `31BD6FE9`=LPO50 Flamethrower, `C86412FF`=Flare gun (+ separate molotovs category)
  - Known open bug: setting a pickup's `fRespawnTime` very low (e.g. 0.1) makes it visually respawn but become permanently un-collectable a second time — unsolved.
- **Named/unique weapon pickups**: e.g. Golden AK47 (`28_pickups.xml`, `Weapons.AK47_new.AK47_Gold`) — set `fRespawnTime` to `0.1` to make it respawn like an armory weapon. To make buddies/mercs wield a named variant, edit `gamemodesconfig.xml`'s `<pack name="buddy">` weapon-assignment section and swap in the variant name (e.g. `AK47.AK47_Gold`).
- **Arms dealer / weapon bazaar** (`gamemodesconfig.xml`, search `cost=`): `availability` (0/1/2 = start/Leboa/Bowa), `needsUnlock` (0/1 = mission-gated), `cost` (must be ≥1 — 0 deletes the entry entirely), `unlockUpgrade` (governs whether the "new items available" message requires owning the base weapon first).
- **Full bandolier/max-ammo reference table** (`gamemodesconfig.xml`, `<!-- BANDOLIER BONUSES -->`): one `<Plan>` per bandolier type, each with per-weapon, per-`difficultyLevel` (0=Casual…3=Infamous) `maxammo` bonuses. Plans confirmed: `pistol_belt` (makarov/star45/deserteagle/6p9), `light_assault_webbing` (mac10/uzi), `shotgun_bandolier` (ithaca/spas12/usas12), `assault_webbing` (mp5/fnfal/g3ka4/ak47/m16), `marksmans_bandolier` (m1903/dragunov/as50/Dart_Rifle), `rocketeer_satchel` (rpg-7/carl_gustaf/mortar), `grenadier_webbing` (m67/m79/ied/MGL140), `pyrotechnic_satchel` (flare_gun/lpo50/molotov), `gunner_pack` (pkm/m249), plus `stealth_camouflage` (camo bonus), `health_basic`/`health_advanced` (healing time + syringe max). This is the complete object-name reference for every carryable weapon/gadget in the base game.
- **Weapon manuals** (`gamemodesconfig.xml`) — three separate bonus systems per weapon: `OPERATIONS MANUALS` (`accuracy`/`damage`/`sticky`[unknown, never resolved]/`recoil`, percent), `REPAIR&MAINTENANCE MANUALS` (`degradation`/`unjamtime`, percent — full default table is `-20%`/`-35%` for nearly every weapon, **except the 6P9 which ships with an unusually strong `-500%` degradation bonus by default**; M-79 and flare gun manuals have no `unjamtime` entry at all since they don't truly jam), `VEHICLE MANUALS` (`degradation`/`repairtime`, percent). Manual bonuses are reflected live in the pause-menu upgrades screen. Setting `degradation` to `-999%` lets you drive an already-"destroyed"/smoking vehicle with no further consequence; setting `unjamtime` to `-99%` skips the unjam animation almost entirely.
- **Enemy loadouts**: `<PrimaryWeapon difficulty="19" probability="0.25" archetype="weapons.Primary.FNFAL" />`-style entries in `gamemodesconfig.xml`'s **`InventoryPacks`** section — swap the `archetype` to rearm enemies with any weapon archetype (including named variants like `AK47.AK47_Gold`), or add a `<SecondaryWeapon probability="X" archetype="…" />` entry (probability is a straight 0.0–1.0 chance, confirmed `1.00` = 100% guaranteed spawn-with). Caveat: enemy AI has no self-preservation logic for player-granted heavy weapons — mercs re-armed with MGL-140s/IEDs frequently kill themselves and teammates with their own ordnance.
- **Complete weapon archetype name reference** (confirmed directly from the game's weapon definitions, useful for any `archetype="weapons.…"` string): Assault rifles — `AK47`, `AK47.AK47_Gold`, `FNFAL`, `G3KA4`, `M16`, `MP5`. Explosives (throwable) — `M67`, `Molotov`, `IED`. Launchers — `Carl_Gustaf`, `Carl_Gustaf.Carl_Gustaf_Merc`, `MGL140`, `M79`, `RPG7`, `RPG7.RPG7_Merc`. LMGs — `M249_Saw`, `M249_Saw.M249_Saw_Merc`, `PKM`, `PKM.PKM_Merc`. Sniper rifles — `AS50`, `Dragunov`, `Dragunov.Dragunov_Merc`, `M1903`, `M1903.M1903_Merc`. Shotguns — `Ithaca`, `SPAS12`, `USAS12`. Special — `Dart_Rifle`, `LPO50`. Pistols/SMGs — `Star45`, `Makarov`, `DesertEagle`, `SilencedMakarov_6P9`, `MAC10`, `Uzi`. **The `_Merc` suffix definitively resolved** (confirmed by stoatoats): it's not a stat variant — it's the *same* weapon redefined as a `Primary` weapon slot instead of `Special`, because mercs have no inventory slot for "special" weapons (this is why, e.g., mercs are seen carrying the AS50 sniper rifle on their backs — it's the `.Merc`-suffixed primary-slot version). Additional variants spotted in world-specific `weapons.xml` files (World 1/2/DLC): mounted-weapon archetypes (`MountedWeapons.M249_FishingBoat`, `MountedWeapons.M249_SandBags.Multi`), "rusty"/degraded pickup variants (`Secondary.MAC10.Mikes_Rusty`), and more special-weapon entries (`Special.Flare_Gun`, `Special.Mortar.Mortar_Merc`) — `28_pickups.xml` is theorized to define these degraded/rusty pickup variants layered on top of the general `WeaponProperties.xml` definitions (unconfirmed but consistent with observed behavior).
- **Faction infighting** (credited to modder StoatOats' "RealMod", reverse-engineered by forum member Vaatho): `10_Ghostpatrols.xml` lists every patrol type with a faction-color field (Blue/Red — the actual values are literally `BlueFaction`/`RedFaction`, confirmed via Discord `🔨-fc2-modding`, Nov 2021) — changing a patrol's color to be hostile toward its nominal "own" side produces real, dynamic inter-faction firefights in the open world (confirmed: wreckage/fire observed after the fact). Patrol vehicles also have an (often empty) passenger slot that accepts an enemy-archetype string — confirmed working values: `Sniper_`, `MortarMan_`, `LightMachineGunner_` — letting you build custom vehicle loadouts. All patrol models default to a "Caucasian" race string, replaceable with "Nubian" for variety. `gamemodesconfig.xml` separately defines a `<ReinforcementArchetypes>`/`<MapArmy>` pair describing a 5×5 grid of which faction "owns" which map region (not the visible 3×3 in-game map division) — early attempts to edit this in isolation showed no effect; the patrol-color method above is what actually worked.
- **Malaria removal**: in `gamemodesconfig.xml`'s `<Malaria>` block, redirect all four curve references (`FirstAttackTime`, `BetweenAttackTime`, `MinorAttackQte`, `MinorAttackDuration`) to `Curves.PlayerSicknessCurves.HealthMax_Infamous` — an unrelated always-high curve, effectively disabling attacks. Confirmed working by multiple independent testers.
- **Camo/detection tuning** (`gamemodesconfig.xml` + `player.xml`): `<Plan name="stealth_camouflage"><bonus attr="stealth" value="1"/></Plan>` controls the camo suit's bonus (base game default is actually `value="10"`, not `1`), seemingly as a 0/1 toggle rather than a gradient. Detection-duration values (in `player.xml`) are **counterintuitive**: raising them makes mercs *slower* to confirm a detection (i.e., harder to detect you), not more alert. Full field list confirmed (guru3D, credit "Freelancer"): `PlayerAwarenessResetDelay`, `LongRangeDetectionDurationLevel1/2`, `MediumRangeDetectionDurationLevel1`, `PersonalRangeDetectionDurationLevel1/2`, `IntimateRangeDetectionDurationLevel1/2`, `StareAtDetectionDurationLevel1/2`, `AimingProvocationDetectionDurationLevel1-4`, `AimedAtDetectionDurationLevel1-3`, `RaisedWeaponLongRangeDetectionDuration`, `RaisedWeaponPersonalRangeDetectionDuration` — plus two unrelated-but-adjacent movement fields in the same block, `BumpMinSpeed`/`ChargeMinSpeed`/`ChargeAngle` (minimum speed to trigger a bump/charge reaction, and the angle threshold for it). Per-biome FOV blocks exist separately for desert/savannah/jungle in `player.xml`'s `SensorySystem/FOVParameters` (`fLength`/`fAngle` for both `FocusFOV` and `PeripheralFOV`), plus global multipliers (`fNightTimeMultiplier` defaults to 0.5 vs. 4 for daytime). Combining biome-specific FOV narrowing with the stealth bonus was the community's most-praised realism tweak (credited to forum member Diablo_Lobo, later folded into the site's knowledge base).
- **Fall damage / high jump**: `fJumpHeight` in `player.xml` (default `1`; `5`+ is already superhuman, values around `20` are common in community patches) combined with `fMinSpeedFallDamage`/`fMaxSpeedFallDamage` in `gamemodesconfig.xml`'s `DefaultCountersService` block (set to e.g. `1000` to eliminate fall damage entirely) — needed together, since high jump without fall-damage removal gets you killed on landing. Crouching instantly reverts to normal movement/jump as a built-in "toggle."
- **Movement speed**: `player.xml`, replace **all instances** (many, with differing defaults) of `fWalkingMaxSpeed`, `fWalkingMaxSpeedCrouch`, `fWalkingAcceleration`, `fWalkingDeceleration`, `archSprintCurve` (references `Curves.Locomotion.Sprint` in curves.xml), `fSprintingDeceleration`, `fClimbSpeed`, `fSwimmingMinDepth`, `fSwimmingMaxSpeed`. A community-tested "fun but extreme" set: walking/swimming speed both at `15`, crouch speed `2.5`, accel/decel `30`/`30`, sprint decel `20`, climb `1.4` — outruns vehicles entirely.
- **Player gravity**: `fGravity` in `player.xml` — setting to a small/negative value (e.g. `-1`) approximates low/moon gravity for glider-less "flying" experimentation. **Vanilla default confirmed as `-20`**, and identical between FC2 and FC3 (Discord, `🔨-fc3-and-bd-modding`, 2023-02-28, from "Low") — useful as a baseline to compare against before tuning.
- **Sniper range**: vanilla `fMaxRange` for sniper rifles is **400** (Discord, `🔨-fc3-and-bd-modding`, 2023-02-28, from "Low", offered while tuning FC3's weapon config) — same source also notes `RangeMultipliers` were not applied on a per-difficulty basis in FC2.
- **Grenade drop chance**: `<ChanceToDropGrenade Casual="1" Experimented="0.5" Hardcore="0.33" Infamous="0.25"/>` in `gamemodesconfig.xml` — set any tier to `1` to make every killed merc drop a grenade.
- **Reputation/Infamy**: `gamemodesconfig.xml`'s `<Infamy><Acts>` block sets per-Act level ranges (`startlevel`/`maxlevel`); `<Levels>` sets `minRate`/`maxRate` bands each with a `medicine` flag, `failureChance`, and `maxNPCInFailure`. **Important behavioral caveat**: directly setting a high starting reputation via save-edit does *not* reproduce the same NPC reactions as organically earning that level through play — the game appears to key some dialogue/behavior off actual earned progress/state, not just the raw displayed number. Relevant for any reputation-driven mod (e.g. a toxic/good-behavior tracking system): don't assume the number alone drives behavior.
- **Weapon sound swapping**: replace a weapon's sound-hash string with another's (concrete confirmed values: dart=`3078303034424635454100`, flare gun=`3078303034353643344200`, silent makarov=`3078303034424635433800`) *and* set `bIsSilent` to `True` together — confirmed to produce a genuinely silent weapon in actual play (mercs don't react to gunfire at range). This contradicts an earlier-generation finding that `bIsSilent` alone "does nothing" — the flag alone is inert, but combined with an actual sound-hash swap it works. Reliability is inconsistent though: swapping AS50↔Flare Gun and FN FAL↔Dragunov-with-fire-mode-change both worked for different testers, but at least one other combination (FN FAL to a different sound while set to automatic) resulted in the weapon losing sound entirely. A `bEmitLight` bool also exists per-weapon; no confirmed visual effect was ever observed from toggling it.
- **Weapon skins/textures**: full texture replacement (not just a color/tint parameter — no such simple recolor mechanism was ever found) is possible and was demonstrated across most base weapons (6P9, MP5, AK47, G3, M249, Dragunov, Uzi, Mac10, SPAS12, USAS12, M16, AS50, mortar) plus UI/loading-screen images, by extracting/editing/repacking texture files through Gibbed's tools. No 3D model/mesh files were ever located or edited by the community — texture-only. **Correction/update from Discord findings**: whole-mesh *swapping between existing characters* was achieved as early as 2022 (see the buddy `.xbg` swap recipe below) and static *object import from another Dunia title* (Avatar) was achieved by 2024–2025 (see [Engine Theory](./engine-theory.md) and the [XBM/XBG format page](../file-formats/xbm-xbg.md)) — "no mesh editing at all" was true of the original 2011–2017 OWG-era community, not of the field as a whole. What genuinely remained unsolved through at least early 2022, confirmed directly ("I don't think anyone has added a model to FC2" — Hunter, Jan 2022): importing a **wholly new, from-scratch custom model** — a 2022-era Noesis-based `.xbg` converter script existed but targeted the wrong XBG sub-version (written for "3/4," FC2 uses "3/2," per Lasercar) and didn't work out of the box. Character-mesh creation was the last major unsolved piece — **resolved as of July 2026** by Quiet_Joker's `Dunia-Engine-XBG-Blender-Importer`, which adds a real (if buggy, pre-alpha) import/export pipeline including the master-skeleton animation mechanism — see the [XBM/XBG format page](../file-formats/xbm-xbg.md) for the full technical explanation and current known bugs. This means "custom weapon/character models in FC2" has gone, within this research's own timeframe, from "no one has ever done this" to "an actively-developed tool exists and mostly works" — worth revisiting this whole bullet's framing once that tool stabilizes.
- **Buddy character model-swap recipe** (cosmetic reskin between *existing* buddies only, not custom import — Discord, `🔨-fc2-modding`, Jan 2022, from Hunter/FC_Redux's author): locate the desired buddy's `.xbg` model inside `worlds.fat`'s `graphics` folder, copy it into the corresponding folder inside `entitylibrarypatchoverride.fcb`, and rename it to match the buddy slot you want to replace (e.g. copy `paul.xbg`, rename to `frank.xbg`, to make Frank appear with Paul's model). Useful for making a specific buddy "available" in an Act where they normally aren't. A `"cannot load world"` error on trying this almost always means `worlds.dat` itself has been moved/deleted from the data folder, not a problem with the swap itself.
- **Old Blender import script exists but is severely limited** (Discord, `🔨-fc2-modding`, Aug 2022): a script-based (not proper add-on) importer only works in **Blender 2.79b specifically** — running it in any newer Blender version crashes. It is **import-only, no export path back into FC2 exists**, and it's missing UV data, rig/skeleton, `.hkx` collision mesh, and animation support entirely — confirmed still-missing pieces as of this conversation, a full year before Gabor/fdx4061's 2025–2026 XBM/XBG work (see the [XBM/XBG format page](../file-formats/xbm-xbg.md)) which approaches the same problem from the opposite direction (binary format parsing rather than a Blender plugin). A practical cross-version workaround discussed: run the old importer in a 2.79b instance, keep a modern Blender open simultaneously, and copy-paste the imported geometry across.
- **Weapon pickup/UI icons are partly hardcoded in `Dunia.dll` itself, not purely data-driven** — a genuinely important architectural exception to the "everything's in FCB/XML" pattern (Discord, `🔨-fc2-modding`, Jul 2022, extensive live investigation by Boggalog/scubrah/Hunter/legendhavoc175): each weapon's HUD icon is selected via its `sName` value, looked up against an icon list; SP uses `hud.mgb` (at `common_unpack\ui\localized\pcwidescreen\eng\ui`), MP uses a separate `hud_mp.mgb` — both `.mgb` files are **not editable by any available tool** ("unworkable"). Machetes (and MP-specific weapons like mounted guns) have **no icon entry at all**, so they fall back to showing the sawed-off shotgun's icon by default. **A working fix was found via direct `Dunia.dll` hex-editing**: repointing an unused internal icon slot (the "dlc6" placeholder) to the machete texture, and reassigning the sawed-off's own `sName` so the fallback no longer collides — this is a second confirmed real-world case (beyond the Multi editor mod's modified `Dunia.dll`, see [Engine Theory](./engine-theory.md)) of the community successfully patching the game **binary**, not just its data files. It wasn't shipped as a finished fix, though: a separate, unrelated bug surfaced where **loading any save always force-equips whichever machete is selected from the main menu**, overriding current loadout regardless of the icon fix — a real dead-end specific to machete/gadget-slot persistence, not the DLL patch itself.
- **Diamond case respawn/quantity data lives in `OA_MissionPickups.xml`** (Discord, `🔨-fc2-modding`, Aug 2022, from "LowPoly"): individual diamond case pickups are defined here, distinguished by a value field — most regular cases use `1`, `2`, or `3`; the player's starting case uses `10`. Directly contradicts an assumption in the same conversation that per-case diamond editing would be "tedious as hell" — it's a single well-structured file, not per-sector hand-editing.
- **Ambient sound regions** (Discord, Far Cry 2 Multiplayer, `modding`, Mar 2026, from Gabor): FC2 has exactly **7 defined ambient sound regions** — `Desert`, `Jungle`, `Savannah` (pure biomes) plus 4 transition/blend regions `tDes_Jun`, `tDesSav`, `tJun_Sav`, `tLake`. Which region plays for a given patch of terrain is driven by **which ground texture is assigned there** — the sound-region name is a property of the texture slot, not the terrain geometry itself, so the practical map-wide lever is reassigning ground textures in the "textures inventory" to ones carrying the sound region you want, then optionally lowering sound intensity. Deeper per-instance control exists via `soundregions.xml`, but changes there are **local-only and don't ship with an exported custom map** — an important distinction for anyone trying to give a custom map a specific ambiance for other players, not just themselves. Untested: whether leaving a sound region assignment empty produces true silence (Gabor's guess: probably, but "too weird" as a design choice).
- **Weapon fields confirmed via live sound/scope modding** (Discord, `modding`, Dec 2025–Jan 2026, from Gabor): `bUseHiResScope` (Boolean, in the weapon's `generated`/entity data, not `41_WeaponProperties.xml`) — toggling to `False` was demonstrated changing scope behavior; `iAnimationValue` reportedly affects which crosshair is used. Re-confirms the already-known "editing the wrong weapon variant is a common silent-failure cause" gotcha in practice — fdx4061 spent real time on an invisible-effect bug before realizing he needed `m16.multi` specifically, not the base `m16` entry.
- **Full-auto conversion + sound looping is its own separate gotcha, distinct from the `iBurstLength`/`selFireRateMode` sound-silencing issue above**: a `.spk` sound authored for a 3-round burst, when used as the fire sound for a weapon modded to genuinely full-auto, was observed to audibly "repeat 3 times with a pause" instead of looping smoothly — the burst-length baked into the sound asset itself doesn't automatically stretch to match a changed fire-rate/mode. The fix demonstrated was manually re-editing the sound to loop seamlessly as a short burst rather than relying on the original burst-length audio.
- **Vehicle armor**: reliability-degradation "manuals" (bought from the arms dealer) pushed toward 100% can make a vehicle nearly immune to mortar/rocket fire while you're inside it (mercs' shots visibly bounce off / do nothing, though bullets were observed passing straight through the windshield into the seats — the "armor" is not a real collision simulation, more a damage-immunity flag); you still take minor chip damage if caught on foot nearby. This bonus was independently confirmed to also reduce damage the *player* takes while inside the vehicle, not just the vehicle's own durability.
- **Mortar/weapon neutering**: setting a weapon's damage/range/accuracy/reliability/firerate all to `0.1` (must be nonzero — `0` still deals damage in at least one tested case) leaves the sound/visual effect intact while making it nearly harmless — used by the community to "de-fang" the mortar-guy NPCs specifically.
