# `.xbm` / `.xbg` — Materials & Meshes

:::note[Community-reported]
Source: Discord, Far Cry 2 Multiplayer, `modding` channel, an extended live reverse-engineering
exchange between **Gabor** (who has his own unreleased XBM↔XML/XBG↔XML converter, "fully fletched,"
built over "years") and **fdx4061** (author of an XBM editor and an XBG texture/material extractor),
April 2026. This is the deepest byte-level format documentation found by the community anywhere —
nothing this granular exists in the OWG threads or any tool README surveyed. Not yet independently
verified by disassembly (see [Getting Started](../modding/getting-started.md) for how RE-verified
and community-reported claims are distinguished across this site).
:::

- **XBM and XBG are structurally the same file format with different extensions** — confirmed directly: "they are the same files in a way, just with a different extension... they have the same structure." An XBM parser can read into an XBG's material data by skipping the XBG-specific mesh (`LTMD`) content, and vice versa the section-parsing logic is shared.
- **FourCC-style section tags, stored reversed in the byte stream**: `HSEM` (= "MESH" reversed, the header/section marker), `EDON` (= "NODE"), `DIKS`/`SULC` (submesh/mesh-block data — "SULC" holds one block per mesh/submesh, each block delimited by `FFFF` marker bytes). This confirms a classic reversed-fourCC chunk format, a strong starting point for a from-scratch parser.
- **Section-parsing algorithm** (this is the actual working algorithm used by both Gabor's and fdx4061's independent tools): after the header, the file is a fixed, always-present sequence of sections in a **constant order**; every section begins with a count of elements it contains. If that count is `0`, the section is empty — skip immediately to the next one. You just walk the sections in order, always reading (and consuming) each one's count first. Confirmed section order for XBM materials: **1) maps** (texture references), **2) reflection** (a single f32), **3) tiling** (2 f32s), **4) colours/RGB** (3 f32s — diffuse base colour), **5) illumination/RGBA** (4 f32s). In practice only one of the colour-RGB or illumination-RGBA blocks is actually populated for a given material even though the format always reserves space for both; the RGBA one is described as "an illumination kind of section." A section reserved for road-texture-only data in FC2 is present but "always empty" except that specific use case, and is used more heavily in Avatar.
- **Field types**: everything in these sections is **`f32`/float32**, *except the very last section, which is `u32`/uint32*. This was an active point of disagreement in the exchange: fdx4061's own tool had converted some values to plain 0–255 RGB integers for a friendlier editing UI, but Gabor pushed back that this **silently clamps modder capability** — the engine's real values can exceed 0–255 (e.g. driving a texture's colour value far above normal makes it visibly glow — useful for tiling/glow effects), so treating them as bounded RGB bytes loses real functionality even if it "still works" for typical cases.
- **String padding rule**: every string in these files is followed by a **1-byte zero-padding**, consistently, in FC2's files specifically — a concrete, reusable parsing rule.
- **XBG-only alignment requirement**: the start of the actual 3D mesh data inside an XBG **must be 16-byte aligned** (offset divisible by 16), or the game breaks when loading it. This does **not** apply to XBM (material) files since they carry no 3D data — some converters (including an early version of Gabor's own) apply the alignment rule unnecessarily to XBM without it mattering either way, "just to do it properly."
- **The character/creature XBG bone-palette mystery — resolved two months later, by a different person.** As of April 2026, neither Gabor nor fdx4061 understood the `SULC` submesh blocks' bone-palette data for character XBGs ("I don't know how bone palettes work... that's why I still can't create a character-type xbg" — fdx4061, verbatim). **By June 2026, Quiet_Joker (author of the `Dunia-Engine-XBG-Blender-Importer`, see [sources](../modding/sources.md)) had cracked the actual mechanism**, discovered "the hard way": an `.xbg` model's bones are only a **local, pruned subset** of a full **master "source skeleton" file** (`.mab`/`.skeleton`) — one shared per model *category* (e.g. all NPCs), containing every bone ever defined for that category at development time, from which each individual model's bones were derived. Critically, **animation does not use the xbg's own local bone order** — because bones get pruned per-model, that local order isn't stable across models — so the engine looks up the master skeleton file as a "master control" reference to know which bones move, then applies that to the local xbg model at animation time. **Three files are required together for a custom/replaced model to animate correctly**: (1) a `.mab` motion/animation-bank file, (2) an `.xbg` that references that animation file, (3) the category's master `.skeleton` reference file. Quiet_Joker notes this bone-inheritance pattern is a known technique from the professional animation industry (cited: Maya), not a Dunia-specific oddity.
- **`Dunia-Engine-XBG-Blender-Importer` (Quiet_Joker) — the actual, working (if still buggy) answer to custom mesh import**, released v3.0 on 2026-07-04 (days before this research pass). Originally built for Avatar: The Game, ported/modernized from a Blender 2.49b-era script lineage (the same one referenced as import-only and 2.79b-locked back in 2022 — see [Data Recipes](../modding/data-recipes.md)'s superseded note); FC2 support is "more or less" working because "Avatar shares the same stuff as far cry 2." **Confirmed working**: static object import, character import (though with broken clothing/UV-tiling material loading in some cases), and a real **export/injection workflow** — import with "Separate Primitives" on, edit or replace geometry in Blender (join in custom meshes, delete old vertices), select only the objects you want written out, then inject/export; the script only writes whatever is currently selected. **Confirmed broken as of this exact conversation**: importing *weapon* XBGs specifically throws a consistent error (reproduced on the AK-47 and a 1911); HKX (Havok collision mesh) export doesn't work yet; a user separately reported not yet getting export to work reliably at all, days after release. FC2 `.xbg` files live in `Data_Win32\worlds\worlds.fat`, not the more obvious `common` archive. The author (who doesn't own every relevant game) is actively developing this against community-submitted broken files — treat as pre-alpha, but real and moving fast.
- **A second, separate, previously-undocumented XBG-injection path was mentioned in passing**: "the only way of importing xbgs back into the game for fc was to use the unreal engine tool made by id-daemon" (Discord, Jul 2026) — id-daemon is independently credited in FCBConverter's own author banner (see [Getting Started](../modding/getting-started.md)). Not verified, named, or linked further in this research — worth tracking down given it predates Quiet_Joker's tool and id-daemon's other credited work is solid.
- **A modder porting the other direction (FC2 → FC3) hit the exact same wall from the opposite side**: "Ganic" ported several FC2 models into FC3 (`avatar_wound_test.xbg` 2023-01-17, `carwrecked01_bk` 2024-03-19, `deserteagle` 2024-03-21, `jackcarver` 2024-07-13 — Discord, `🔨-fc3-and-bd-modding`). Concrete blockers reported: *"xbt itself is not a problem, but there is no xbm converter"* and *"I cannot make a full-fledged xbg from scratch, therefore I am limited in the indexes of materials for each part in one xbg"* (2024-03-20) — i.e. raw geometry porting works, but there's no clean `.xbm` (material) converter and rebuilding an `.xbg` from scratch fails on material indices/bone weighting, the same unsolved edge of this format from the FC3 side that Gabor/fdx4061/Quiet_Joker were working from the FC2 side above.
- **A cross-game "skeleton reading tool" (fdx4061), named `SkeleTree`** (shared 2025-10-06, now preserved in `research/reference-files/tool-archives/`), confirmed to work across **Avatar (2009), Far Cry 2, and Far Cry 3** — direct evidence these three titles share a compatible skeleton/rig format at the binary level, not just the broader Dunia lineage already established.
- **`.spk` filenames are themselves hashes** (e.g. `004492b8.spk`, `004492b6.spk`), consistent with the hash-based naming already established for FCB/archive content — see [the `.spk` format page](./spk.md).
